Kickoff Rule

Do not over-collect before starting.

You only need enough to confirm the paid scope and begin. The minimum package is transaction hash, brief, and any relevant links.

Kickoff Path

Use the same path every time after deposit.

1. Confirm Payment

Ask for the transaction hash and confirm that the deposit landed on the correct wallet and network.

2. Collect Brief

Get the project brief in the shortest usable form. Do not turn it into a long discovery phase.

Open Brief Board

Fast Intake Option

If the sprint is already clear, use the minimum intake page instead of forcing the client through a fuller brief.

3. Start Sprint

Confirm scope in one short message and begin the sprint deliverables.

4. Handle Repeat Questions

If the client still asks about timing, revisions, or payment route, send the FAQ page instead of rewriting answers.

What To Send

Keep the client request small and explicit.

Standard Request

After payment, send:
tx hash
project brief
any relevant page links

That is enough for me to begin.

Short Version

Send the tx hash after payment and the project brief. I will take it from there.

If They Ask What Happens Next

After payment:
1. you send the tx hash
2. you send the project brief
3. I confirm scope
4. I start on the sprint deliverables

Starter Scope

Use the same core framing at kickoff.

Current Leak

Leads can request a quote or make contact without getting a fast next step, so warm intent cools while the team is busy.

What Ships

One follow-up message flow, one intake summary structure, one quote-prep or callback handoff, and one owner-facing SOP.
