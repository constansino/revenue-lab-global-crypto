Referral monthly-decision page

Use monthly support only when the next value really repeats after the referral sprint.

This page is for the moment after a closed referral win when the same buyer may need ongoing support. The key decision is whether the next value is recurring optimization and upkeep, or a fresh project-shaped bottleneck. Keep the conversion clean: confirm the repeating need, package the monthly rule, rebuild the approval path, and route into the first funded month without muddying the first sprint.

Conversion Rule

A referral retainer should begin when the same need is repeating, not when a second sprint is still undefined.

Good retainer conversion is a choice, not a default. It reuses trust from the first referral step, but still separates recurring upkeep from larger new build work and keeps the first month commercially clean.

When To Convert

Convert to monthly work when continuity creates more value than another isolated sprint.

Optimization Repeats

Best when the next wins come from small recurring improvements around the system already installed.

Upkeep Is Predictable

Best when the buyer now needs maintenance, review, or tuning on a regular rhythm instead of another concentrated rebuild.

Approval Can Be Simple

Best when the first sprint created enough trust that one compact recurring yes is realistic.

Not A Hidden New Sprint

If the real need is a new large bottleneck, forcing monthly packaging too early will just create recurring confusion.

Same Buyer Context

Best when the monthly work is for the same buyer and system, not when the real opportunity is channel scale across more buyers.

Fresh Payment Fits

A real conversion should still produce a fresh first-month payment event, not rely on leftover commercial momentum.

Short Lines

Use direct language that makes the monthly conversion legible.

Need Line

The next value looks recurring, so the clean move may be one defined monthly rhythm rather than another project.

Boundary Line

If a larger new bottleneck appears, I want to split that into a fresh sprint instead of hiding it inside the retainer.

Approval Line

Even though the first step is done, the recurring rule should still go through one clean approval path before the first month starts.

Payment Line

The start of monthly support should have its own first-month payment and proof trail just like any other real cycle.

Do Not Do This

Most weak retainer conversions happen when recurring work is treated as a softer version of the sprint.

Do Not Convert Too Early

If the first sprint is still commercially fuzzy, the recurring model will inherit that ambiguity.

Do Not Hide New Build Work

A big new deliverable should be sold as a new sprint, not a justification for vague recurring access.

Do Not Skip The Approval Reset

Monthly support still needs one clean buyer-side yes before the first recurring payment lands.

Do Not Blur First-Month Payment

The first month should feel like a new commercial event, not like leftover money from the last sprint.
